## 1. 功能目标
提供一个安全、高效的待办事项管理 REST API，允许用户通过 JWT 认证后对自己的待办事项进行创建、查看、标记完成和删除操作，确保用户数据严格隔离，并遵循统一的错误处理规范。

## 2. 用户故事
| 角色 | 行为 | 期望结果 |
|------|------|---------|
| 新用户 | 使用用户名和密码注册 | 成功创建账号，返回用户信息（密码不回传） |
| 已注册用户 | 使用用户名和密码登录 | 获取有效的 JWT Token |
| 已登录用户 | 创建待办事项（标题必填，描述和截止日期可选） | 待办创建成功，返回包含 id 的待办对象 |
| 已登录用户 | 查看自己的待办列表（支持分页） | 收到按创建时间倒序排列的待办列表及分页信息 |
| 已登录用户 | 查看自己的单个待办详情 | 返回该待办的完整信息 |
| 已登录用户 | 标记自己的待办为已完成 | 待办状态变为已完成，返回更新后的对象 |
| 已登录用户 | 删除自己的待办 | 待办被删除，返回 204 |
| 已登录用户 | 尝试访问他人的待办 | 返回 403 Forbidden |
| 未登录用户 | 访问任何待办接口 | 返回 401 Unauthorized |

## 3. 功能点
### 3.1 用户注册
- **描述**：用户通过用户名和密码注册新账号。
- **输入**：用户名（string，必填）、密码（string，必填）。
- **输出**：成功返回 201 和用户信息（id, username），失败返回错误码及错误信息。
- **前置条件**：无。
- **后置条件**：新用户记录创建，密码加密存储。

### 3.2 用户登录
- **描述**：已注册用户通过用户名和密码登录，获取 JWT Token。
- **输入**：用户名（string，必填）、密码（string，必填）。
- **输出**：成功返回 200 和 JWT Token（string）；失败返回 401 及错误信息。
- **前置条件**：用户已注册。
- **后置条件**：服务端生成 Token，客户端需在后续请求的 Authorization 头携带（Bearer Token）。

### 3.3 创建待办事项
- **描述**：认证用户创建一条新的待办事项。
- **输入**：标题（string，必填）、描述（string，可选）、截止日期（date，可选）。
- **输出**：成功返回 201 和待办对象（id, title, description, due_date, completed=false, created_at, updated_at, user_id）。
- **前置条件**：用户已认证（请求携带有效 JWT）。
- **后置条件**：新待办记录创建并与当前用户关联。

### 3.4 获取待办事项列表
- **描述**：认证用户获取自己的待办列表，支持分页，按创建时间倒序。
- **输入**：查询参数 – page（可选，默认 1）、page_size（可选，默认 20）。
- **输出**：成功返回 200 和待办列表数组，以及分页元数据（total, page, page_size, total_pages）。
- **前置条件**：用户已认证。
- **后置条件**：无。

### 3.5 获取待办事项详情
- **描述**：认证用户获取单个待办的详细信息。
- **输入**：路径参数 – todo_id。
- **输出**：成功返回 200 和待办对象；若 todo_id 不存在或不属于当前用户则返回 404 或 403。
- **前置条件**：用户已认证。
- **后置条件**：无。

### 3.6 标记待办事项为已完成
- **描述**：认证用户将某个自己的待办状态标记为已完成。
- **输入**：路径参数 – todo_id。
- **输出**：成功返回 200 和更新后的待办对象（completed=true）。
- **前置条件**：用户已认证，待办存在且属于当前用户。
- **后置条件**：待办记录更新 completed 字段为 true。

### 3.7 删除待办事项
- **描述**：认证用户删除自己的某个待办。
- **输入**：路径参数 – todo_id。
- **输出**：成功返回 204 无内容；若不存在或不属于当前用户返回 404 或 403。
- **前置条件**：用户已认证，待办存在且属于当前用户。
- **后置条件**：待办记录从数据库删除。

## 4. 业务规则
- 用户名必须唯一，不区分大小写时视为相同。
- 密码需经过加密存储（如 bcrypt）。
- JWT Token 中必须包含用户 ID，并设置过期时间（建议 24 小时）。
- 所有待办接口（创建、列表、详情、完成、删除）均需认证，通过验证 Authorization 头中的 Bearer Token。
- 用户只能操作自己的待办事项；访问他人待办时返回 403 Forbidden。
- 统一错误响应格式：`{ "code": 状态码, "message": "描述信息" }`。
- 创建待办时标题不能为空，否则返回 400。
- 分页参数 page 最小为 1，page_size 最大为 100，超出范围返回 400。
- 截止日期（若提供）需为有效的 ISO 8601 日期格式（如 2025-01-01），否则返回 400。
- 平均响应时间应小于 200ms（合理数据量下）。
- 系统需支持 10,000 并发用户。

## 5. 边界条件
| 场景 | 预期行为 |
|------|---------|
| 注册时用户名已存在 | 返回 409 Conflict |
| 登录时用户名或密码错误 | 返回 401 Unauthorized |
| 未携带 Token 访问受保护接口 | 返回 401 Unauthorized |
| Token 无效或已过期 | 返回 401 Unauthorized |
| 访问不存在的待办 ID | 返回 404 Not Found |
| 访问他人的待办 | 返回 403 Forbidden |
| 创建待办时标题为空 | 返回 400 Bad Request，message 提示标题必填 |
| 创建待办时截止日期格式无效 | 返回 400 Bad Request，提示日期格式错误 |
| 分页 page 为 0 或负数 | 返回 400 Bad Request，提示 page 必须 >=1 |
| 分页 page_size 超过 100 | 返回 400 Bad Request，提示 page_size 最大 100 |
| 删除不存在的待办 | 返回 404 Not Found |
| 标记已完成一个已完成的待办 | 视为成功，返回 200，completed 仍为 true |
| 并发修改同一个待办（如同时完成和删除） | 原子性保证，一个成功其余失败，返回适当错误（如 409） |
| 大量并发请求（10,000 并发） | 系统正常运作，平均响应时间 <200ms |

## 6. 验收标准
- [ ] 用户可注册新账号（用户名唯一），返回用户信息且密码不回传
- [ ] 用户可登录并获得有效 JWT Token
- [ ] 未提供 Token 时访问待办接口返回 401
- [ ] 使用有效 Token 可创建待办（标题必填，描述和截止日期可选），返回 201 和待办对象
- [ ] 创建待办时标题为空返回 400
- [ ] 查看待办列表支持分页参数（page, page_size），默认 page=1、page_size=20，按创建时间倒序，返回分页元数据
- [ ] 查看单个待办详情返回正确信息；不存在返回 404；他人的返回 403
- [ ] 标记自己的待办为已完成，返回 200，completed 变为 true
- [ ] 删除自己的待办返回 204；不存在返回 404；他人的返回 403
- [ ] 所有错误响应均为 `{ "code": int, "message": string }`，包含 400、401、403、404、409、500 等
- [ ] 平均响应时间 <200ms（正常负载下）
- [ ] 支持 10,000 并发用户（压力测试通过）

## 7. 不做什么（Out of Scope）
- 用户角色管理（管理员、普通用户等）
- 待办事项的编辑（修改标题/描述/截止日期）
- 待办事项的优先级、标签、分类
- 待办事项搜索功能（仅提供列表分页）
- 待办事项协作/共享
- 退出登录（Token 注销）及 Refresh Token 机制
- 忘记密码 / 重置密码
- 修改用户信息（用户名、密码等）
- 国际化和多语言
